🚀 新平台接入CRM计提接口文档

完整接口说明 · 字段映射 · 请求示例 · 响应处理

版本：v1.0 | 更新时间：2026年7月

📋 一、接口概述

接口名称 计提数据推送接口
请求方式 POST
接口地址 http://{CRM域名}/crm/api/o/provision/external/accept
Content-Type application/json
认证方式 签名验证（MD5）
调用方向 新平台 → CRM

📝 二、请求参数

2.1 外层参数结构（ExternalParamBo）

参数名 类型 必填 说明
sign String 必须 签名字符串（每次请求不同）
timestamp Date 必须 请求时间戳
count Integer 必须 data数组的长度
data List<CrmData> 必须 计提数据列表

2.2 计提数据字段（CrmData）

参数名 类型 必填 说明 示例值
contractCode String 必须 合同号（CRM中已存在） HT202607001
country String 必须 国家 US
media String 必须 媒体 Google
provisionMoney BigDecimal 必须 计提收入（结算币种） 10000.00
origUsdMoney BigDecimal 必须 原始预计收入（美金） 1200.00
usdMoney BigDecimal 必须 预计收入（美金） 1200.00
earningType String 必须 收入类型 收入
contractType String 必须 合同类型（0-收款，1-付款） 0
settlementMonth String 必须 结算月份 2026-07-01 00:00:00
remark String 可选 备注 7月计提
accrualBy String 可选 计提人 张三

⚠️ 重要说明

  • 不需要传的字段：id、errorReason（CRM会自动生成provisionId）
  • contractCode必须存在：CRM中必须有这个合同，否则会报错"合同不存在"
  • country+media唯一：同一合同+月份下，country+media必须唯一
  • 日期格式：settlementMonth格式必须是 yyyy-MM-dd HH:mm:ss

🔑 三、签名算法

3.1 签名生成规则

// 1. 准备参数 Map<String, Object> param = new HashMap<>(); param.put("data", data); // 计提数据（JSON数组） param.put("secretKey", APP_SECRET); // 固定密钥 param.put("timestamp", timestamp); // 时间戳 param.put("count", count); // 数据条数 // 2. 排序并拼接 // 按key排序，拼接成: key1=value1&key2=value2&...&secretKey=xxx // 3. MD5加密 String sign = MD5(拼接后的字符串).toUpperCase();

3.2 固定密钥

public static final String APP_SECRET = "244e7f76db84b915ff10dd9d239eeb4b";

🔑 签名特点

  • 每次请求都不同：因为timestamp和data内容都在变化
  • 安全性：保证请求未被篡改
  • 幂等性：相同参数+时间戳产生相同签名

📥 五、响应参数

5.1 响应结构

参数名 类型 说明
code String 状态码
messageInfo String 提示信息
data List<Pair<String, String>> 失败的记录（key=id, value=错误原因）

5.2 状态码说明

状态码 含义 说明
200 成功 所有数据处理成功，data为空
4002 部分成功 部分数据失败，data中包含失败记录
400 失败 所有数据处理失败

5.3 响应示例

✅ 成功响应

{ "code": "200", "messageInfo": "success", "data": [] }

⚠️ 部分成功响应

{ "code": "4002", "messageInfo": "部分数据处理失败", "data": [ { "key": "HT202607001-2026-07-01 00:00:00-US-Google", "value": "合同不存在" } ] }

❌ 六、错误码说明

错误码 错误原因 解决方案
400 签名验证失败 检查签名算法和密钥是否正确
400 数据为空 检查data数组是否为空
400 参数数量有误 检查count是否与data数组长度一致
400 合同不存在 检查contractCode是否在CRM中存在
4002 部分失败 查看data中的具体错误原因

🔄 七、接入流程

步骤1：确认合同已在CRM中创建
步骤2：按照CrmData格式准备计提数据
步骤3：实现签名算法（MD5）
步骤4：构造请求参数（ExternalParamBo）
步骤5：发送HTTP POST请求
步骤6：处理响应结果
步骤7：失败数据重新推送

✅ 注意事项

  • 合同必须存在：推送前确保contractCode在CRM中已创建
  • 数据唯一性：同一合同+月份+国家+媒体不能重复推送
  • 失败重试：失败的数据可以重新推送，CRM会自动更新
  • 批量推送：可以一次推送多条数据，建议每批不超过100条

🆕 八、CRM生成的字段

💡 关键说明

以下字段由CRM自动生成，新平台不需要传递：

字段 生成规则 说明
id 雪花算法 数据库主键，19位数字（用户不可见）
provisionId 合同号+月份 格式：{合同类型前缀}{contractCode}{settlementMonth}

provisionId格式示例

收款合同

S0042272026-06

  • S = 收款合同（Sale）
  • 004227 = contractCode
  • 2026-06 = settlementMonth

付款合同

P0042282026-07

  • P = 付款合同（Purchase）
  • 004228 = contractCode
  • 2026-07 = settlementMonth

✅ 九、总结

📋 新平台需要做的事

  • ✅ 准备数据：按照CrmData格式准备计提数据
  • ✅ 实现签名：按照签名算法生成sign
  • ✅ 发送请求：POST到CRM接口
  • ✅ 处理响应：根据状态码处理成功/失败
  • ✅ 失败重试：失败的数据可以重新推送

❌ 新平台不需要做的事

  • ❌ 不需要生成provisionId（CRM自动生成）
  • ❌ 不需要生成id（CRM自动生成）
  • ❌ 不需要关心计提平台的数据库
  • ❌ 不需要关心计提平台的内部逻辑
  • ❌ 不需要传递id字段

🔑 关键字段清单

必传字段 说明

settlementMonth 结算月份（格式：yyyy-MM-dd HH:mm:ss）
country 国家
