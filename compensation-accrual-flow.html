二、步骤详解

步骤 0：检查直客账户

目的：确定本月是否有直客账户需要赔付计提，为步骤 1 的数据拉取提供账户列表

数据来源：赔付管理页面（直客账户 Tab）

处理逻辑：


1. 从赔付管理页面读取本月标记为「需要赔付」的直客账户列表
2. 如果列表不为空：
   → 步骤 1 拉取数据时，需同时拉取这些直客账户的赔付数据
   → 直客账户的核代账户ID 在数鲸数据中为 -1 或空
   → 后续步骤 4/5 中，直客账户的聚合和匹配使用 account_id 代替核代ID
3. 如果列表为空：
   → 步骤 1 只拉取代理账户数据
   → 直客相关逻辑跳过

赔付账户管理（赔付管理页面）：

账户类型默认行为可配置
代理账户默认全部需要赔付✅ 可排除部分核代ID（配置为不赔付）
直客账户默认不赔付✅ 需手动标记是否需要赔付

操作方式：

- 在赔付管理页面维护需要排除的核代ID列表

- 拉取数据时直接排除这些核代ID

- 每月初由运营确认本月的赔付范围

步骤 1：拉取数鲸原始数据

目的：获取每个账户在每个国家×媒体维度的赔付基础数据

前置判断：

1. 从赔付管理页面读取本月需要赔付的账户列表

- 代理账户：默认全部需要赔付，但可支持部分代理不赔付（在赔付管理页面配置排除）

- 直客账户：需手动标记是否需要赔付

2. 拉取数据时，直接排除配置为「不赔付」的核代ID

3. 如果有直客账户需要赔付，拉取时需同时包含代理账户和直客账户的数据

数据来源：数鲸平台

- 链接：https://s.mi.cn/6mH3c0

- 取数范围：当月1日至T-1日的累计数据（T为开始计提的日期）

- 例：6月18日开始计提 → 取6月1日-6月17日的数据

- 取的是累计总数，不是单日数据

数据维度：

- 代理账户：核代账户ID × 账户ID × 国家(ipcountry) × 媒体ID(media_id)

- 直客账户：账户ID × 国家(ipcountry) × 媒体ID(media_id)（核代账户ID 为 -1 或空）

拉取字段：

字段说明
核代账户id核算代理账户 ID（一个核代下可能有多个账户）。**直客账户此字段为 -1 或空**
account_id实际广告账户 ID
ipcountry国家编码（如 DE, ES, CL, OT）
media_id数鲸的媒体 ID（数字编号）
赔付金_0时区该维度的赔付金额（0 时区）
赔付口径广告主价值_0时区该维度的广告主价值（0 时区）
dsp_fee该维度的 DSP 收入

系统自动计算参数：

- 本月取几天 = T-1（开始计提日的前一天日期数字）

- 例：6月18日开始计提 → 本月取几天 = 17（拉取6月1日-6月17日的数据，共17天）

- 由赔付计提的开始时间决定，无需手动填写

- 本月有几天 = 系统根据当前月份自动判断

- 例：6月 → 30天，2月 → 28/29天，1月/3月/5月等 → 31天

- 由系统自动获取，无需手动填写

时间折算公式：


赔付金额 = 当前取值 / 本月取几天 × 本月有几天
例：取了17天数据，当月30天 → 赔付金额 = 当前值 / 17 × 30

计算公式：


超成本金额（每行）= dsp_fee - 广告主价值
  即：DSP收入 - 广告主价值
  如果 > 0 → 需要赔付
  如果 ≤ 0 → 赔付金额为 0

账户超成本总额（每行）= 该账户下所有国家×媒体的 dsp_fee 总和 - 广告主价值总和
  用途：计算每行在账户总超成本中的占比

超成本比例（每行）= 该行超成本金额 / 该账户超成本总额
  用途：按比例分摊赔付金额

账户实际赔付金额（每行）= 超成本比例 × 该账户赔付金总额 × (本月有几天 / 本月取几天)
  即：按比例分摊 + 时间折算
  例：取了17天数据，当月30天 → 时间折算 = 30/17

国家*媒体实际超成本金额 = 账户实际赔付金额（后续用于匹配CRM底表）

校验（两个独立数据源对比）：

校验的目的是确认按国家/媒体维度拆分后的赔付金额与账户维度的赔付金额是否一致（差异来源于四舍五入）。


数据 A — 账户维度赔付金额（从数鲸直接拉取）：
  拉取方式：同一数鲸链接，按账户ID维度拉取 T-1 的赔付金额
  预估全月 = 账户赔付金额 / T-1 × 当月天数

数据 B — 账户*国家*媒体维度赔付金额（本步骤计算得出）：
  = 按上述比例分摊 + 时间折算后的所有行赔付金额之和

diff = 数据A的预估全月总额 - 数据B的赔付总额
日均diff = diff / 当月天数
  → 日均diff ≤ 100 美金：正常（四舍五入误差）
  → 日均diff > 100 美金：需人工核查（可能存在数据异常）

差异来源：

- 按国家/媒体维度拆分后，每行四舍五入保留两位小数

- 重新聚合后与账户维度的原始值产生微小差异

- 日均 diff 通常在 50 美金左右，属正常范围

步骤 2：国家和媒体名称替换

目的：将原始数据中的国家编码和媒体ID替换为计提平台使用的标准名称

核心逻辑：


输入：账户 × 国家编码 × media_id 维度的赔付金额
  ↓
国家替换：部分国家编码 → OT（如 CL→OT, 其他非标准国家→OT）
媒体替换：media_id → 计提媒体名称（如 317→海外小米音乐, 7963→海外联盟业务-SDK）
  ↓
输出：账户 × 国家名称 × 媒体名称 维度的赔付金额

处理思路（推荐方案）：

将计提配置文件增加 media_id 列，实现一步替换：


计提配置文件增加 media_id 列后：
  media_id | media_type | 计提媒体名称
  317      | MSA        | 海外小米音乐
  7963     | GLOBAL_MEDIA_7963 | 海外联盟业务-SDK
  ...

赔付数据中的 media_id → 直接通过配置文件匹配 → 得到计提媒体名称
不需要额外跑数据工厂的 SQL

当前实现方式（两步替换）：

第一步：获取 media_id 与 media_type 的对应关系

- 数据工厂链接：https://dt.mi.com/workspace/#/workspace/11188/adHoc/17012541

- SQL 查询：


  SELECT DISTINCT media_id, media_type
  FROM iceberg_alsgprc_hadoop.ad_dim.dim_global_tag_pid_df

- 说明：从数据工厂维表拉取 media_id 与 media_type 的对应关系，每月跑一次

- 与运营预估规则一致：赔付的国家/媒体替换规则与运营预估模块的替换规则相同

MSA 媒体按比例拆分

MSA（media_type=MSA）不直接替换为单一媒体名称，而是按分成系数拆分到多个子媒体：

原始媒体media_type拆分到媒体分成系数
MSAGLOBAL_368海外安全中心0.24
MSAGLOBAL_BROWSER海外浏览器0.05
MSAGLOBAL_VIDEO海外小米视频0.05

MSAGLOBAL_THEME_MANAGER海外主题0.09
MSAGLOBAL_DOWNLOAD_MANAGER海外下载管理器0.03

拆分计算：


某子媒体赔付金额 = MSA原始赔付金额 × 该子媒体分成系数
例：MSA赔付 $10,000 → 海外安全中心 = $10,000 × 0.24 = $2,400
所有子媒体拆分金额之和 = 原始MSA赔付金额
分成系数总和 = 1.00

- 替换国家列不能有 NA（未匹配的国家）

- 替换媒体列不能有 NA（未匹配的媒体）

- 有 NA → 需要特殊处理（手动补充映射或删除该行）

步骤 3：获取运营预估的计提数据 + 汇率

目的：获取运营在计提平台完成的预估数据，作为赔付扣减的基数

数据来源：计提平台导出（运营预估完成后的数据）

数据维度：核代账户 × 合同ID × 国家 × 媒体

字段：

字段说明
核代账户供参考，推送CRM时删除
合同IDS+6位（如 S004075）
国家已替换后的标准国家名
媒体已替换后的标准媒体名
美金金额（系统金额）运营预估的美金金额（原始计提金额）
计提金额（结算币种）美金金额 × 汇率（结算币种金额）
收入类型预估收入
结算月份如 2026-05
结算币种USD / EUR / INR 等

汇率配置：

- 在计提平台的汇率配置页面维护

- 每月更新一次

- 用于美金 → 结算币种的换算

步骤 4：按核代维度聚合赔付数据

目的：将步骤 1 的细粒度赔付数据按核代维度聚合，便于与 CRM 底表匹配

操作：对步骤 1 的数据做数据透视表

透视维度：

- 代理账户：核代账户ID × 替换国家 × 替换媒体

- 直客账户：账户ID × 替换国家 × 替换媒体（因为直客的核代ID为空/-1，聚合到 account_id 上）

聚合指标：国家*媒体实际超成本金额（求和）

匹配名构造：

- 代理账户：核代账户ID + 国家 + 媒体（字符串拼接）

- 例：368BRShareMe（核代368 + 巴西 + ShareMe）

- 直客账户：account_id + 国家 + 媒体（用 account_id 代替核代ID）

- 例：515BRShareMe（直客账户515 + 巴西 + ShareMe）

特殊处理：

- 直客账户 ID 替换：如 515 → 4799（手动处理，后续迭代自动化）

- 需要排除的核代ID已在步骤 0 的赔付管理页面配置，拉取数据时已排除

步骤 5：匹配赔付金额 + 计算赔付后金额（核心步骤）

目的：将赔付金额匹配到每条计提数据上，计算赔付后的计提金额

匹配逻辑：


构造 CRM 底表的匹配名：
  代理账户 = 核代账户ID + 国家 + 媒体
  直客账户 = account_id + 国家 + 媒体（直客核代ID为空/-1，用account_id）

用匹配名去透视表（步骤4）中 VLOOKUP 赔付金额

逐行计算公式：


K列 - 匹配名 = A列(核代账户) + C列(国家) + D列(媒体)

L列 - 国家*媒体实际超成本金额
  = VLOOKUP(K列, 透视表匹配名范围, 赔付金额列, 0)
  即：通过匹配名从透视表中找到对应的赔付金额

M列 - 美金金额减超成本金额
  = E列(美金金额/系统金额) - L列(赔付金额)
  即：运营预估金额 - 赔付金额

N列 - 去掉小于0的，保留2位小数
  = IF(M列 < 0, 0, ROUND(M列, 2))
  即：如果赔付后金额为负数则置0（避免倒赔），保留两位小数

O列 - 减去赔付后计提金额（结算币种）
  = N列 × 汇率
  即：赔付后美金金额 × 结算汇率 = 结算币种金额

校验字段：


P列 - 是否重复匹配（校验一个核代ID是否有俩合同）
  = IF(COUNTIF(A列范围, 当前核代账户ID) > 1, "重复", "")
  如果同一核代ID对应多个合同 → 标记告警，需运营确认

Q列 - 合同匹配名 = B列(合同ID) + C列(国家) + D列(媒体)
  用途：与推送CRM时的匹配名对比

Y列 - 算出来的赔付总额 = SUM(O列)
Z列 - 赔付原始表赔付总额 = 步骤1的赔付总额
AA列 - diff = Y列 - Z列
  → diff 应该等于没匹配上CRM的金额
  → 如果 diff 比没匹配上的大 → 说明有核代ID重复合同导致重复赔付

三、异常处理

异常场景处理方式
日均 diff > 100 美金人工核查，常见原因：四舍五入、账户核代关系变更
同一核代ID对应多个合同标记告警，运营确认（通常业务上合并为一个）
赔付后金额 < 0置 0（避免倒赔）
媒体名称 NA（未匹配）手动补充映射或删除该行
直客账户手动处理（合同号映射如 515→4799），后续迭代
CRM 推送失败联系赵赫排查，常见原因：合同未双签、行数限制
合同未审批通过本月不计提该合同

二、步骤详解

步骤 0：张真更新主表

目的：更新所有核代账户的预估收入数据（已扣除赔付金额）

负责人：张真

操作：

1. 张真完成主表更新（包含所有核代的收入数据）

2. 上传到共享文档

3. 推送到工作群

数据维度：核代账户 × 国家 × 媒体

数据内容：

- 核代账户的预估收入（已扣除赔付金额）

- 分媒体的收入明细

- 包含海外联盟媒体的收入数据（后续需扣除）

步骤 1：运营复制核代扣除赔付后收入

目的：获取自己负责的核代账户的收入数据

操作：

1. 从张真的文档中找到自己负责的合同

- 例：合同3012 对应核代4799

2. 复制该核代的「扣除赔付后收入」数据

3. 粘贴到返点计算工作表

数据维度：核代账户 × 国家 × 媒体

步骤 2：查询返点系数

目的：获取当前月份的返点系数

数据来源：数金平台（由销管维护）

查询方式：

1. 打开数金链接（分区域面板：亚太组/中东/欧洲等）

2. 选择所在区域的面板

3. 查看最新月份的盘点数据

4. 找到「总计返点」= 代理基础返点 + 动态X返点

返点系数：

客户返点系数说明
客户31321613%该客户所有合同统一系数
客户123410%该客户所有合同统一系数
......每个客户系数可能不同

计算方式：

- 可按单个合同单独计算

- 也可按同一客户合并计算（同一客户下所有合同按统一系数）

步骤 3：计算返点基数

核心公式：


返点基数 = 扣除赔付后收入 - 扣除项

扣除项（所有区域统一规则，各区域一致）：

扣除项说明处理方式
海外联盟媒体收入（offline）所有区域均扣除从返点基数中减去
非标资源收入如果非标在单独合同中则不影响；如果与标准在同一个合同则需要扣除按合同维度判断

不需要扣除的项：

- MSA 媒体：不需要去除，保留在返点基数中

- 区域差异：目前各区域规则统一，无区域特殊扣除

计算示例：


核代4799 扣除赔付后收入：$499,000
其中海外联盟媒体收入：$94,994
非标资源收入：$12,000（出海区域扣除）

返点基数 = $499,000 - $94,994 - $12,000 = $392,006

步骤 4：计算预估返点金额

核心公式：


预估返点金额 = 返点基数 × 返点系数

计算示例：


返点基数：$392,006
返点系数：13.5%
预估返点金额 = $392,006 × 13.5% = $52,921

多客户计算：


客户313216（返点系数13%）：
  核代A：基数 $200,000 × 13% = $26,000
  核代B：基数 $150,000 × 13% = $19,500
  合计：$45,500

客户1234（返点系数10%）：
  核代C：基数 $300,000 × 10% = $30,000

步骤 5：填入上传模板

操作：

1. 将计算结果填入上传模板

2. 金额为 0 的条目必须删除（否则上传失败）

3. 保存

模板字段：

字段说明
合同IDS+6位
核代账户ID—
国家—
媒体—
返点基数（USD）扣除海外联盟等后的基数
返点系数如 13.5%
预估返点金额基数 × 系数
结算月份当月

步骤 6：上传CRM

操作：保存后打开CRM确认上传成功

校验：上传后检查CRM中的数据是否与本地计算一致

三、返点基数规则汇总（各区域统一）

MSA 媒体**不需要去除**，保留在返点基数中
区域差异目前各区域规则统一
