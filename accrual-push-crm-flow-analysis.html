🎯 核心流程说明

计提平台完成计提计算后，将数据通过HTTP接口推送到CRM系统。CRM系统验证签名、解析数据，并写入计提管理表。整个过程支持批量推送、部分失败处理和状态追踪。

四、签名机制

4.1 签名生成逻辑

🔑 关键点

签名每次都不一样！因为签名包含了时间戳、数据内容和数据条数，这些参数每次请求都在变化。

// SignUtil.java public static String generateSignSign(Object Data, Date date, Integer count) { Map<String,Object> param = new HashMap(16); param.put("param", Data); // ⭐ 数据内容（每次不同） param.put("secretKey", APP_SECRET); // 固定密钥 param.put("timestamp", date); // ⭐ 时间戳（每次不同） param.put("count", count); // ⭐ 数据条数（每次可能不同） return createSign(param, APP_SECRET); } // 签名算法 public static String createSign(Object param, String accessSecret) { // 1. 将参数转为JSON再转为Map // 2. 对Map的key进行排序 // 3. 拼接字符串：key1=value1&key2=value2&...&secretKey=xxx // 4. MD5加密后转大写 String sign = DigestUtils.md5Hex(temp.toString()).toUpperCase(); return sign; } // 固定密钥（计提平台和CRM系统共用） public static final String APP_SECRET = "244e7f76db84b915ff10dd9d239eeb4b";

4.2 签名组成要素

要素 是否固定 说明
data 不固定 计提数据内容，每次推送的数据不同
timestamp 不固定 请求时间戳，精确到毫秒
count 可能变化 数据条数，取决于当月未推送的数据量
secretKey 固定 APP_SECRET，计提平台和CRM共用

五、id字段说明

5.1 id的来源

💡 关键点

id在计提保存到数据库时就已生成，推送时直接使用。推送过程中不会再生成新的id。

// AccrualMiadsAutomation实体类 @TableId(type = IdType.AUTO) // ⭐ 数据库自增主键 @ApiModelProperty(value = "id") private Integer id;

5.2 id的生命周期

1️⃣ 计提保存

用户点击保存按钮

数据写入数据库

2️⃣ 数据库生成id

accrual_miads_automation表

自增主键：12345

3️⃣ 推送到CRM

crmData.setId("12345")

使用已有的id

4️⃣ CRM接收

验证id是否存在

写入CRM计提表

// 推送时id已经存在 // AccrualMiadsAutomationServiceImpl.java crmData.setId(item.getId() + "") // item.getId() = 12345（数据库主键） // CRM端接收 // ProvisionAcceptBo.java private String id; // 接收计提平台传来的id

5.3 为什么需要id？

  • 唯一标识：用于标识每条计提记录
  • 幂等性：防止重复推送同一条数据
  • 失败重试：推送失败后可以重新推送，CRM通过id判断是否已存在
  • 状态追踪：记录每条数据的传输状态

六、同步机制详解

6.1 计提平台推送流程

// DataTransmissionServiceImpl.java @Override public Result<?> sendDataToSRM() { // 1. 获取当月和上月时间 List<Integer> thisAndLastMonth = DateUtils.getThisAndLastMonth(); int thisMoth = Integer.parseInt(String.valueOf(thisAndLastMonth.get(0))); // 2. 查询所有未传输数据 List<AccrualMiadsAutomation> miadsData = iAccrualMiadsAutomationService.recordQueryForCRM(thisMoth); List<AccrualOfflineAutomation> offlineData = iAccrualOfflineAutomationService.recordQueryForCRM(thisMoth); List<AccrualPreinstallAutomation> preinstallData = iAccrualPreinstallAutomationService.recordQueryForCRM(thisMoth); // 3. 字段映射转换为CrmData List<CrmData> crmData = new ArrayList<>(); crmData.addAll(iAccrualMiadsAutomationService.crmFieldMapping(miadsData)); crmData.addAll(iAccrualOfflineAutomationService.crmFieldMapping(offlineData)); crmData.addAll(iAccrualPreinstallAutomationService.crmFieldMapping(preinstallData)); // 4. 检查是否有数据 if (crmData.isEmpty()) { return Result.error(CommonMsg.NO_DATA_THIS_MONTH); } // 5. HTTP调用CRM接口 ResponseDataFromCrmVO responseData = sendHttpRequesttoCRM(crmData); // 6. 处理返回结果 String code = responseData.getCode(); String state; switch (code) { case ("200"): // 全部成功 state = CommonConstant.STATE_0; saveDataTransmission(CommonConstant.STATE_0); break; case ("4002"): // 部分成功 state = CommonConstant.STATE_2; List<String> failIds = responseData.getData().stream() .map(row -> row.getKey()).collect(Collectors.toList()); saveDataTransmission(CommonConstant.STATE_2, String.join(",", failIds)); break; default: // 全部失败 saveDataTransmission(CommonConstant.STATE_1); throw new AutomaticAccrualException(responseData.getMessageInfo()); } // 7. 更新传输状态 iAccrualMiadsAutomationService.updateTransmissionState(miadsData, CommonConstant.STATE_0); iAccrualOfflineAutomationService.updateTransmissionState(offlineData, CommonConstant.STATE_0); iAccrualPreinstallAutomationService.updateTransmissionState(preinstallData, CommonConstant.STATE_0); return Result.OK(resMap); }

6.2 HTTP请求发送

// DataTransmissionServiceImpl.java private ResponseDataFromCrmVO sendHttpRequesttoCRM(List<CrmData> crmDatas) { Date date = new Date(); Object obj = JSON.parse(JSON.toJSONString(crmDatas)); // ⭐ 生成签名（每次不同） String sign = SignUtil.generateSignSign(obj, date, crmDatas.size()); Map<String, Object> params = new HashMap<>(); params.put("data", obj); params.put("sign", sign); params.put("timestamp", date); params.put("count", crmDatas.size()); String json = JSON.toJSONString(params); log.info("推送CRM数据: {}", json); // 发送HTTP POST请求 try (HttpResponse response = HttpRequest.post(CRM_URL) .header("Content-Type", "application/json") .body(json) .execute()) { if (response.getStatus() != HttpStatus.HTTP_OK) { throw new AutomaticAccrualException("CRM系统返回错误码:" + response.getStatus()); } body = response.body(); } // 解析CRM返回结果 responseDataFromCrmVO = JSON.parseObject(body, new TypeReference<ResponseDataFromCrmVO>() {}); return responseDataFromCrmVO; }

七、状态管理

7.1 传输状态码

状态码 常量 含义 说明
0 STATE_0 传输成功 数据已成功推送到CRM
1 STATE_1 传输中/待传输 数据尚未推送或正在推送中
2 STATE_2 传输失败 推送失败，需查看失败原因

7.2 CRM返回码

返回码 含义 处理方式
200 全部成功 所有数据写入CRM，状态更新为STATE_0
4002 部分成功 返回失败的ID列表，成功的更新为STATE_0，失败的保持STATE_2
400或其他 全部失败 所有数据状态更新为STATE_2，记录错误原因

7.3 失败处理

⚠️ 失败场景

  • 合同不存在：CRM中找不到对应的合同
  • 签名验证失败：sign不匹配，可能是密钥错误或数据被篡改
  • 网络异常：无法连接CRM系统
  • 数据格式错误：字段类型不匹配或必填字段缺失
  • 重复推送：同一条数据已经存在于CRM中

十、常见问题

❓ Q1: 为什么签名每次不一样？

A: 签名算法中包含了时间戳(timestamp)、数据内容(data)和数据条数(count)，这些参数每次请求都在变化，所以生成的签名也会不同。这是为了保证请求的唯一性和安全性。

❓ Q2: CrmData的id字段是推送时生成的吗？

A: 不是。id在计提数据保存到数据库时就由数据库自增主键生成了。推送时直接使用这个已有的id，用于CRM端识别和去重。

❓ Q3: 什么情况下会自动推送到CRM？

A: 只有满足以下条件时才会自动推送：
1. 业务类型是Miads（程序化）
2. 一级DSP = "online"
3. 保存计提时触发
其他情况都需要手动点击"传输"按钮。

❓ Q4: 推送失败后如何处理？

A: 系统会记录失败原因，并返回Excel文件供用户查看。用户可以：
1. 查看失败原因（合同不存在、签名错误等）
2. 修正问题后重新推送
3. 通过"传输"按钮重新推送失败的数据

❓ Q5: 可以推送历史月份的数据吗？

A: 默认情况下（accrual.crm.month=true）只能推送当月数据。如果需要推送历史数据，需要修改配置文件中的month为false。
