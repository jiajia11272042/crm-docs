一、需求背景与目标

背景

IIB-CRM 已与 miads 打通产品准入审核链路（LLC → 法务 → RSA）。部分品类在部分国家属于禁投品类，但广告主申报的品类不一定准确，审核人员在更正品类（所属行业）时无法直观知道该品类在哪些国家禁投，容易错判放行。

本需求要求 CRM 在产品准入审核的各个环节，将禁投信息实时提示给审核人员，并在审批提交时自动驳回命中禁投的国家，从而把合规要求落到系统里。

ℹ️
合规属性：这是合规要求，全量生效，不做灰度、不做 A/B test。

目标

  • 审核三环节（单条详情页 / 批量审批弹窗 / 修改审核结果弹窗）均能展示禁投品类与禁投国家，禁投国家红色高亮。
  • 切换「所属行业（品类）」后，禁投提示实时联动刷新。
  • 审批「通过」时，命中禁投的（产品-国家）记录自动驳回，驳回原因自动写入「XX品类在XX国家禁投」；批量场景下未命中的行正常通过。
  • 提供历史数据批量驳回工具：品类被新增进禁投配置后，可一键清理历史已通过的禁投国家。

影响方

核心功能影响方影响点
品类-国家限制配置运营在 SSP 平台 Abram 配置中心维护「品类 → 禁投国家」映射
产品准入审核本地化(LLC)、法务、RSA 审批人命中禁投品类后，对应国家不可审核通过

二、总体业务流程

📌
关键数据模型前提（贯穿全文）：miads 送审的每条产品记录只对应「一个国家」。已核对代码：miads 入站 DTO MiadsInboundProductRequestVo.targetRegion 是单个字符串；审批列表查询每行 bp.target_country as country 也是一个国家。因此——
  • 单条审核 / 修改审核结果（需求一、三）：每个详情只有一个国家，命中判断退化为二元：命中禁投 or 不命中，不存在"部分国家命中"。无需国家高亮列表、无需"通过其余国家"。
  • 批量审批（需求二）：勾选多条，但每一行仍是一个产品 + 一个国家（见批量弹窗表格的 countryName 单数列）；多国家只在"多行"层面体现，不在单行内。
  • （注：详情页 qualificationVos 表按 targetCountry.split(",") 渲染，理论支持多国，但 miads 链路下实际恒为一条。）

三、禁投配置说明

项说明
配置来源SSP 平台 Abram 模块 配置中心，由运营创建「品类-国家限制规则」
数据源SSP Abram 配置是唯一数据源。CRM 直接读取 SSP 这份配置；miads 读取的也是同一份配置，CRM 不经 miads 中转
规则格式品类ID（riskIndustryId） → 禁止投放的国家/地区编码列表（如 ["CN","IN","BR"]）
生效方式配置发布后，CRM 实时读取，无需重启或同步任务
回滚通过 Abram 配置中心回滚或修改后重新发布
🆕
SSP 接入需新开发（CRM 此前从未对接过 SSP 平台）。需新增一套 SSP client，工作量包括：
  • 新增 SSP API client：在 xm-api 模块下新建（如 SspAbramApi），参照现有 BusinessAuditApi / MiAdsApi 的接入范式（Feign/HTTP + 鉴权头）。
  • 新增配置项：各环境 application-*.yml 增加 SSP 的 base_url / app_key（或 token）等，向 SSP/运维申请鉴权凭证与各环境地址。
  • 封装统一 service 方法 getRestrictCountriesByIndustry(List<riskIndustryId>)：返回每个品类的禁投国家编码，供所有禁投查询复用。
  • 缓存：建议加短期本地缓存（如 5min），降低审核页频繁查询对 SSP 的压力。
  • 容错：SSP 不可用时降级——返回空、隐藏禁投提示、记录日志，不阻断审核主流程。

四、数据结构与接口总览

4.1 禁投信息：独立拉取（不进产品详情接口）

⚠️
设计决策：禁投信息单独拉取，不塞进 lookProductApprove。产品信息由 miads 送审而来，本身不含禁投信息；禁投配置在 SSP Abram，是另一数据源。参照页面现有「商店审核结果」（storeInfo 侧边栏走独立接口 /l/productApprove/storeInfo、独立 loading、失败不阻断详情）的做法，禁投信息也独立拉取。
① 解耦：SSP 慢/挂时详情页主体不受影响，禁投提示异步补上、失败仅隐藏；
② 天然复用：审批人切换品类时本就要按新品类重查禁投，所以首屏展示与切换联动用同一个接口 checkRestrict，无需 lookProductApprove 再冗余返回一份。
// POST /l/product/checkRestrict —— 首屏 + 切换品类 均调此接口（单产品=单国家） // 入参 { "riskIndustryId": 1024, "country": "CN" } // 返回 { "isHitRestrict": true, // 该品类在该国家是否禁投（二元） "hitRestrictIndustry": "赌博游戏", // 命中禁投的品类名称 "countryName": "中国" // 该国家中文名（用于提示文案） } // 批量场景 POST /l/product/batchDetail 入参为多行，逐行判断： // [{ "productId":1, "riskIndustryId":1024, "country":"CN" }, ...]

判断口径：命中与否取审核人更正后的品类 correctedRiskIndustryId；若未更正则取原始 riskIndustryId。因每条产品仅一个国家，命中是二元的（该品类在该国是否禁投）。前端调用时机：详情页 getList() 拿到产品后调一次（首屏），handleIndustryChange 切换品类后再调（联动）——与 fetchStoreInfo 同级的独立异步调用。

需求一 · CRM 产品审核详情页提示禁投品类和国家 P0

涉及页面：/approveManagement/productDetail.vue

场景说明：单条产品审核详情页，该产品只对应一个目标国家。页面顶部「产品信息」区的「所属行业」为只读字段；审批人在页面下方独立的「LLC 行业更正」区切换品类（仅 LLC/法务角色可见）。禁投命中判断按「当前生效品类（更正后 ?? 原始）」+ 该国家进行，二元命中。

本需求覆盖两种命中情况，分别对应下方两个原型：

  • 情况一：进页面即命中 —— 提交的原始品类在该国家就是禁投的（不依赖更正动作就要提示）；
  • 情况二：审批人更正后命中 —— 原始品类不禁投或品类被更正为禁投品类，提示在 LLC 更正区下方实时刷新。

需求逻辑

场景触发条件页面表现
A. 原始品类即命中进页面，原始 riskIndustryId + 该国家命中禁投「所属行业」下方红色提示；目标国家字段红色高亮；「通过」按钮拦截
B. 更正后命中LLC 在更正区选新行业后命中更正区下方红色提示；目标国家字段红色高亮
C. 更正后未命中原始命中、更正为不禁投的行业隐藏禁投提示，目标国家恢复常态
D. 原始与更正均未命中当前生效品类在该国家不禁投不显示禁投提示

  • 不改 lookProductApprove 返回结构；新增独立接口 POST /l/product/checkRestrict，按 correctedRiskIndustryId ?? riskIndustryId + 产品的单个国家调 SSP。

  4. 批量更正行业（可选）：顶部选统一行业后，所有选中行改用该行业重新判断命中并刷新；不选则各自沿用 correctedRiskIndustryId ?? riskIndustryId。
  5. 历史审核品类提示：保留现有 historyRiskIndustryList 展示，仅作辅助参考，不参与禁投命中计算。
  6. 批量通过：命中行自动驳回（原因「XX品类在XX国禁投」），未命中行通过。批量驳回：全部驳回，原因以填写为准。
实现说明：现有批量审批是前端循环逐条调 approveUpdate（见 processBatchApprove）。本期保持该模式：batchDetail 一次性返回每行的品类与命中结果用于展示；提交时由后端在单条 update 内按该产品当前生效品类完成禁投命中行的自动驳回（前端把命中国家随参数带上，或后端按 productId 重新计算）。

五、边界与异常处理

情形处理
读取禁投配置接口超时/失败不阻断审核页加载；隐藏禁投提示并记录日志（查询失败不影响展示，与现有 getHistoryRiskIndustryInfo 容错一致）
产品无目标国家 / 国家为空不展示禁投提示（理论上 miads 送审必带一个国家）
品类未命中（单国家）不显示禁投提示，正常通过——单国家场景无"部分命中"中间态
二级品类命中判断按实际选中的品类ID查询配置（兼容三期二级品类结构 parent_id）
批量：选中行品类各异逐行用各自品类判断禁投，命中行单独驳回，互不影响；提交时逐条处理，单行失败不阻断其他行
批量：顶部更正了统一行业所有选中行改用新行业重新判断命中后再提交
历史审核品类提示仅辅助参考，不参与禁投命中计算（命中以当前生效品类为准）
编辑已送审数据已提交的限制国家正常展示，保存时不校验已提交的，仅校验新增的限制国家

六、FAQ

Q1：品类限制国家配置在哪里？

在 SSP 平台 Abram 模块配置中心创建「品类-国家限制规则」。

Q2：配置如何生效？

配置发布后，CRM 实时读取生效（建议加短期缓存）。

Q3：已审核的限制国家如何处理？

是否刷历史数据待业务确认。方案已在「需求四 · 历史产品批量驳回」中完整设计（手动触发工具），确认后即可落地。

Q4：配置错误怎么办？

通过 Abram 配置中心回滚，或修改后重新发布。
